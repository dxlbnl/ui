# B19: Package documentation

## Context

B19 is the publish-gate item for @dxlbnl/ui. Once the component layer (B1–B14) is
confirmed stable, B19 formalises the library surface with three artefacts:

1. A consumer-facing `README.md` covering installation, peer deps, global CSS, palette
   setup, a minimal usage example, and a Storybook link.
2. A `CHANGELOG.md` in Keep a Changelog format documenting everything delivered by
   B1–B14 under version `0.1.0`.
3. JSDoc `@param` / `@default` annotations on every exported component's props, so
   Storybook autodocs can generate a prop table without a separate documentation pass.
4. A version bump in `package.json` from `0.0.1` to `0.1.0`, signalling "this is a
   usable public build."

Relevant wiki pages: [vision.md](../vision.md), [architecture.md](../architecture.md),
[requirements.md](../requirements.md), [backlog.md](../backlog.md).

The existing `README.md` was generated by the Vibin seed repo and documents the
scaffolding workflow — not the design system. B19 overwrites it completely with
library-consumer documentation.

No `CHANGELOG.md` exists at the project root before this item. `package.json` currently
declares version `0.0.1`.

## Acceptance criteria

### File presence

1. `README.md` exists at the project root after the implementation.
2. `CHANGELOG.md` exists at the project root after the implementation.

### README.md — required sections and content

3. `README.md` contains an H1 heading of `@dxlbnl/ui` (exact text; the seed repo
   heading `# Vibin` must be gone).

4. `README.md` contains an installation code block with exactly the command
   `pnpm add @dxlbnl/ui`. An npm/yarn equivalent may follow but the pnpm line must
   be present and correct.

5. `README.md` lists the peer dependencies. At minimum these two must appear, with their
   version ranges:
   - `svelte`: `^5.0.0`
   - `@sveltejs/kit`: `^2.0.0`

   Verification: `grep -n "svelte" README.md` returns lines covering both entries.

6. `README.md` contains the global CSS import instruction. The import path must be one
   of the following (either is acceptable; the spec does not mandate one over the other):
   - `import '@dxlbnl/ui/dist/index.css'`
   - `import '@dxlbnl/ui/styles.css'`

   The import must appear in a code block (fenced with triple backticks). It must be
   accompanied by prose explaining where to put it (e.g. `src/app.css` or
   `+layout.svelte`).

   > Open question OQ-1 determines which exact path to use — see below.

7. `README.md` contains a palette activation snippet. It must show how to add
   `data-palette="phosphor"` (or `"paper"`) to the `<html>` element, or alternatively
   reference importing the CSS token file and the role of `data-palette`. The snippet
   must explain both palette names.

8. `README.md` contains a basic usage example as a fenced code block. The example must:
   - Import at least one component from `@dxlbnl/ui` (e.g. `Button`).
   - Use that component in Svelte template syntax.
   - Be syntactically valid Svelte 5.

9. `README.md` contains a Storybook link. Because Storybook is not yet deployed publicly
   the link may reference `localhost:6006` or a placeholder URL, but the text must
   mention "Storybook" and the `pnpm storybook` command must be shown or referenced.

### CHANGELOG.md — format and content

10. `CHANGELOG.md` begins with the heading `# Changelog` (exact text, H1).

11. `CHANGELOG.md` contains a version heading for `0.1.0`. The format must be:
    `## [0.1.0] - YYYY-MM-DD` where `YYYY-MM-DD` is a real date (not a placeholder).

12. The `0.1.0` section contains an `### Added` sub-heading with a bullet list of
    entries. There must be at least one bullet per backlog item in B1–B14 (14 bullets
    minimum). Each bullet must name the feature or component(s) delivered by that item.
    Acceptable groupings: one bullet per backlog item, or one bullet per logical feature
    grouping (e.g. "Layout primitives: Stack, Inline, Spread, Grid, Container, Rule").

    Verification: `grep -c "^-" CHANGELOG.md` returns 14 or more.

13. `CHANGELOG.md` follows the Keep a Changelog convention
    (https://keepachangelog.com/): the `[0.1.0]` heading uses square-bracket version,
    and the file ends with a link-reference block defining `[0.1.0]`. The link may point
    to a GitHub compare URL or to `#` if the repo is not public.

### package.json — version bump

14. `package.json` field `"version"` is exactly `"0.1.0"`.

    Verification: `node -e "const p = JSON.parse(require('fs').readFileSync('package.json','utf8')); process.exit(p.version === '0.1.0' ? 0 : 1)"` exits 0.

### JSDoc annotations on exported component props

The following components are exported from `src/lib/index.ts` and must each have JSDoc
annotations on their custom props (those defined in the component's own `interface Props`
— not inherited HTML attributes from `HTMLButtonAttributes`, etc.):

**Primitives** (`src/lib/components/primitives/`):
- `Button.svelte` — `as`, `variant`, `href`
- `Led.svelte` — `color`, `blink`
- `TagPill.svelte` — `variant`
- `Text.svelte` — `variant`, `color`, `as`
- `Heading.svelte` — `level`, `variant`, `color`, `as`

**Layout** (`src/lib/components/layout/`):
- `Stack.svelte` — `as`, `gap`
- `Inline.svelte` — `as`, `gap`, `align`
- `Spread.svelte` — `as`
- `Grid.svelte` — `cols`, `gap`
- `Container.svelte` — `maxWidth`
- `Rule.svelte` — no custom props beyond HTML attrs; annotation is optional

**Cards** (`src/lib/components/cards/`):
- `Card.svelte` — `as`, `href` (if present)
- `ProductCard.svelte` — `name`, `tagline`, `price`, `href`, `image` (or equivalent props)
- `ProjectCard.svelte` — `title`, `description`, `href`, `tags` (or equivalent props)
- `NoteCard.svelte` — `id`, `title`, `date`, `excerpt`, `href` (or equivalent props)

**Navigation** (`src/lib/components/navigation/`):
- `Nav.svelte` — `links`, `siteName`

**Forms** (`src/lib/components/forms/`):
- `Input.svelte` — `error`, `size` (or equivalent custom props)
- `Textarea.svelte` — `error` (or equivalent)
- `Select.svelte` — `options`, `value`, `placeholder`, `error` (or equivalent)
- `InputWrap.svelte` — `prefix`, `suffix`, `clearable` (or equivalent)
- `Field.svelte` — `label`, `hint`, `error`, `inputId` (or equivalent)

**Feedback** (`src/lib/components/feedback/`):
- `Modal.svelte` — `open`, `title`, `variant`, `onclose`

**Patterns** (`src/lib/components/patterns/`):
- `Alert.svelte` — `variant`, `title`, `message`
- `CtaBlock.svelte` — custom props (title, body, href, or equivalent)
- `StatCard.svelte` — custom props (label, value, unit, trend, or equivalent)
- `KvList.svelte` — custom props (items or equivalent)
- `ProgressBar.svelte` — `value`, `max`, `variant` (or equivalent)
- `ActivityRow.svelte` — custom props (status, label, time, or equivalent)
- `SectionHead.svelte` — custom props (title, subtitle, or equivalent)
- `SectionFoot.svelte` — custom props (or equivalent)
- `PageHero.svelte` — custom props (title, lede, or equivalent)

**Data** (`src/lib/components/data/`):
- `Accordion.svelte` — no custom props beyond `children`; annotation is optional
- `AccordionItem.svelte` — `title`, `open` (or equivalent)
- `Tabs.svelte` — `tabs`, `active`, `variant`
- `Table.svelte` — `headers`, `rows` (or equivalent)

15. Every file listed above that has one or more custom props in its `interface Props`
    must contain at least one `@param` or `@default` JSDoc tag in the component's
    `<script>` block.

    Verification: For each component file, `grep -l "@param\|@default" <file>` returns
    the file name (i.e. the annotation is present).

16. The JSDoc annotations use the format accepted by Storybook autodocs for Svelte 5
    `$props()`. The annotation block is placed immediately above the `let { ... } =
    $props()` destructure line (or above the `interface Props` declaration). The minimum
    required format is:

    ```svelte
    /**
     * @param {Type} propName - Description of the prop.
     * @default defaultValue
     */
    let { propName = defaultValue, ...rest }: Props = $props()
    ```

    Alternatively, if JSDoc is placed on the `interface Props` fields using `/** ... */`
    inline comments above each field, that is also acceptable:

    ```ts
    interface Props {
      /** Description of the prop. @default 'primary' */
      variant?: ButtonVariant
    }
    ```

    Either placement is acceptable as long as the annotation is machine-readable by
    Storybook's documentation extractor.

17. Every prop that has a default value in the `$props()` destructure must have a
    `@default` tag. Props with no default (required props, e.g. `title: string`) must
    have a `@param` description but no `@default` tag.

    Verification: For each component, a `grep -c "@default"` in the file returns a count
    equal to or greater than the number of props with a `= value` default in the
    `$props()` line.

18. Annotation density: at least 80% of all `.svelte` files under `src/lib/components/`
    that are exported from `src/lib/index.ts` must contain at least one `@param` or
    `@default` JSDoc tag after implementation.

    Verification:
    ```bash
    total=$(grep -rl "from '\./components" src/lib/index.ts | wc -l)
    annotated=$(grep -rl "@param\|@default" src/lib/components/**/*.svelte | grep -v stories | wc -l)
    # annotated / total >= 0.8
    ```
    (Approximate — exact script may vary. The criterion is ≥ 80% coverage.)

## Out of scope

- **npm publish**: B19 does not publish the package. The version bump to `0.1.0` is
  purely a metadata change.
- **Auto-generated docs**: no TypeDoc, no JSDoc-to-HTML generation. All doc artefacts
  are hand-authored.
- **Storybook deployment**: the `pnpm build-storybook` static output and any CDN/hosting
  are not part of B19.
- **Detailed API reference prose**: that is B21's scope. B19 covers the minimum for a
  consumer to install and use the library.
- **Stories**: no new `.stories.svelte` files. JSDoc feeds Storybook autodocs passively;
  no new stories are added in B19.
- **`docs/` directory**: the machine-readable markdown docs in `docs/` are B21's scope.
- **`@deprecated` or `@since` tags**: not required. Only `@param` and `@default` are
  in scope.
- **Inherited HTML attributes**: JSDoc is only required on props defined in the
  component's own `interface Props`. Passthrough attrs from `HTMLButtonAttributes` etc.
  do not need annotation.
- **Components not yet built** (B15–B20 items): if B15–B18 components exist in the
  codebase at the time B19 runs, they should also be annotated — but they are not
  required by this spec, which targets B1–B14 exports only.

## Open questions

### OQ-1 (blocking for AC-6): What is the correct global CSS import path?

The `package.json` `exports` field currently exposes only `"."` (the JS entry point).
There is no `"./styles.css"` or `"./dist/index.css"` export entry. For AC-6 to be
testable the implementer must either:

- Add `"./styles.css": "./dist/index.css"` (or similar) to the `exports` map in
  `package.json`; or
- Determine the exact dist path that `svelte-package` emits for `app.css` / `tokens.css`.

**Impact**: if this is not resolved, the CSS import instruction in `README.md` will be
incorrect and consumers will not be able to apply design tokens. This is a genuine
blocker for a consumer-ready README.

**Recommendation**: the implementer should run `pnpm prepack` (dry-run build), inspect
the `dist/` output, and add the correct `exports` entry before writing the README.

### OQ-2 (non-blocking): Should `pnpm add @sveltejs/kit` be listed as a peer dep?

`package.json` lists only `svelte: ^5.0.0` as a peer dep. Some components import from
`$app/environment` (Nav uses `browser`). This means consumers must be running SvelteKit
(or provide the `$app/environment` shim). If `@sveltejs/kit` is not declared as a peer
dep, consumers using plain Svelte (not SvelteKit) will get a runtime error.

**Recommendation**: add `"@sveltejs/kit": "^2.0.0"` to `peerDependencies` and list it
in `README.md`. If the project is later decoupled from SvelteKit (removing the
`$app/environment` import from Nav), the peer dep can be removed.

### OQ-3 (non-blocking): Storybook autodocs JSDoc format compatibility with Svelte 5.

Storybook 10 + `@storybook/addon-svelte-csf` reads JSDoc from Svelte component source.
The exact placement that produces the autodocs table (above `interface Props` vs. above
`$props()` vs. inline field comments) should be verified against a real autodocs render
before the implementation is committed. If inline field comments are not picked up,
the `@param` block above `$props()` is the fallback.

**Recommendation**: the implementer should test JSDoc placement on one component (e.g.
`Button.svelte`) and confirm the autodocs table renders the annotation before annotating
all components.
